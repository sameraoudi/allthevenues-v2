How the shortlist works
• Save (heart): every venue card's heart becomes a real toggle. Tapping it adds/removes the venue in localStorage (persists across visits on the device — no account needed) and updates the header Shortlist N count live. All handled in the self-hosted app.js (CSP-safe, no inline JS).
• /shortlist page: the header "Shortlist" link carries the saved ids (/shortlist?ids=1,2,3); the server renders those published venues with the standard venue card. Removing one (heart) drops it from localStorage + the page and updates the Enquire CTA.
• One enquiry: "Enquire about these venues" hands off to the EXISTING multi-venue enquiry flow (/enquire?venues=1,2,3) — one enquiry record linked to all selected venues via enquiry_venues; the user gets one confirmation. No schema change needed.
• Remove + Undo: removing a venue shows a brief toast — "Removed from shortlist. Undo" — that re-adds it if tapped (auto-dismisses after a few seconds).
• Max size: the shortlist caps at 12 venues; past that, adding shows a gentle "Shortlist full" hint instead of saving.
• Graceful handling: the server renders only published venues from validated ids; any saved venue that's since unpublished/archived/deleted is skipped and pruned from localStorage (with a small "some saved venues are no longer available" note). When the shortlist is empty, the "Enquire about these venues" CTA is hidden and the empty state shows.
• Continuity: the multi-venue enquiry form (/enquire?venues=…) lists the selected venue names before submission, so the user sees exactly what they're enquiring about.
• Clear shortlist is a soft, separated text link (not beside the primary CTA) to avoid accidental clearing. Empty state offers Browse venues + Get help finding a venue.
• This preview shows the populated + empty states + a toast stacked; the real page shows one at a time. Cards reuse the standard venue-card partial (gradients here are placeholders).
